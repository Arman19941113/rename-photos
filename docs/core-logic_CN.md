# 核心逻辑说明

本文档整理了 Rename Photos 当前实现中的主要运行逻辑，面向需要维护项目的开发者，重点说明文件如何进入应用、如何生成预览文件名、如何执行重命名，以及冲突在什么阶段被处理。

## 运行结构

- React 前端负责交互流程和用户状态。
- Tauri Rust 后端负责文件系统访问、文件过滤、元数据提取和最终重命名。
- 前后端通过 3 个 Tauri 命令通信：
  - `get_files_from_dir`
  - `get_files_from_paths`
  - `rename_files`
- 这些命令以 async Tauri command 暴露，并把阻塞式文件系统工作交给 Tauri 的 blocking task runtime 执行，因此目录读取、元数据提取和批量重命名不会直接运行在命令调用线程上。

主要入口：

- 前端应用入口：`src/App.tsx`
- 文件操作主逻辑：`src/hooks/useFiles.ts`
- 拖拽监听：`src/hooks/useDragging.ts`
- 模板输入事件：`src/components/OperationBar.tsx`
- 模板输入与历史记录：`src/hooks/useInputFormat.ts`
- 持久化设置服务：`src/services/storage.ts`
- 目标文件名生成：`src/util/file-transformer.ts`
- 通用日期/路径辅助函数：`src/util/common.ts`
- 后端文件命令：`src-tauri/src/commands/file.rs`
- 后端文件辅助逻辑：`src-tauri/src/utils/file/*.rs`

## 文件进入应用的流程

应用支持两种输入方式：

### 1. 打开文件夹

- 前端调用系统目录选择器。
- 选中的目录会传给 `get_files_from_dir`。
- 后端只扫描该目录的直接子项，不会递归扫描子目录，并返回所有适合展示到 UI 的文件。
- 如果扫描到的子路径无法表示为 UTF-8，后端会让整个目录读取失败，而不是静默跳过该条目。

### 2. 拖拽文件或目录

- 前端监听 Tauri 的拖拽事件，并把拖入路径交给 `get_files_from_paths`。
- 如果用户只拖入了一个非符号链接目录路径，后端会把它当作 `get_files_from_dir` 处理。
- 符号链接目录会走常规 UI 排除规则，并作为快捷方式/符号链接被跳过，而不是被打开。
- 否则，后端会把输入视为“明确指定的一组文件路径”，并对每个输入路径应用常规 UI 排除规则，因此目录和其他应排除路径都会被忽略。
- 对于明确指定的文件列表，只要其中任一路径的文件系统 metadata 无法读取，整批加载就会失败，而不是返回部分成功结果。EXIF/媒体 metadata 提取失败不会让加载失败，只会在对应文件上填充 `metaError`。

这个区别会直接影响重名冲突：

- “打开文件夹”会把该目录下的直接文件加载进应用。
- “拖入若干文件”只会加载这些被拖入的文件。
- 同目录下那些没有拖入的文件，不会出现在前端预览集合里。

## 后端的文件过滤规则

后端构建文件列表时，会先排除不应出现在 UI 中的项：

- 文件夹
- Windows `.lnk` 快捷方式
- Unix 符号链接
- 通过读取 `com.apple.FinderInfo` 扩展属性并检查 Finder alias 标志位判断出来的 macOS Finder alias
- 隐藏文件：Unix/macOS 按文件名 `.` 前缀判断，Windows 按 hidden 文件属性判断
- Windows 系统文件：按 system 文件属性判断

这些过滤都发生在元数据提取之前，也发生在前端拿到文件列表之前。

## 文件分类与元数据

每个通过过滤的路径都会先被分类为以下三种之一：

- `image`
- `video`
- `other`

分类逻辑位于 `src-tauri/src/utils/file/kind.rs`，依据是文件扩展名。

图片扩展名：

- `jpg`、`jpeg`、`png`、`webp`、`tif`、`tiff`、`heic`、`heif`、`heics`、`avif`
- 类 RAW 格式：`dng`、`arw`、`cr2`、`cr3`、`nef`、`raf`、`rw2`、`orf`、`sr2`、`srf`、`pef`、`x3f`

视频扩展名：

- `mp4`、`mov`、`m4v`、`3gp`、`avi`、`mkv`、`webm`、`mka`

其他通过过滤的路径都会被归类为 `other`。

元数据提取采用 best-effort 模式：

- 图片通过 `src-tauri/src/utils/file/image_exif` 使用 Rust `exif` crate。
- 标准图片容器会优先读取；如果失败，后端会继续尝试已支持的 RAW 兼容 fallback。
- Panasonic RW2 文件会先尝试读取内嵌 JPEG EXIF；如果失败，再在内存里把 RW2 的 TIFF header 规范化，并把完整文件数据交给 `exif` 解析。
- 视频使用 `nom_exif`。
- 如果提取失败，文件仍然会返回，只是 `metaError` 会被填上。
- 如果提取成功但字段不完整，返回的 metadata 可能是部分字段。
- `other` 类型文件只会展示基础信息，不带元数据。

运行时元数据字段：

- 图片暴露 `date`、`make`、`camera`、`lens`、`focalLength`、`aperture`、`shutter`、`iso`。
- 视频暴露 `date`、`make`、`camera`。
- 图片字段来自 EXIF tag：`DateTimeOriginal`、`Make`、`Model`、`LensModel`、`FocalLengthIn35mmFilm`、`FNumber`、`ExposureTime`、`PhotographicSensitivity`。
- 视频字段来自 `nom_exif` track tag：`CreateDate`、`Make`、`Model`。
- 图片日期直接使用 `exif` 读取 `DateTimeOriginal` 后的显示值，常见格式是 `YYYY:MM:DD HH:mm:ss`。
- 视频日期在 `nom_exif` 返回可解析的 local/RFC3339 值时会转成本地时区的 `YYYY-MM-DD HH:mm:ss`；否则保留原始值。
- 如果解析成功但没有任何当前追踪字段，后端会返回 `metadata: null`，且不会填充 `metaError`。

前端会把后端返回值转换成 `UIFile`，并计算元数据状态：

- `SUCCESS`：存在元数据，且当前追踪的字段都齐全
- `WARNING`：元数据缺失一部分，或者整体为空
- `ERROR`：后端解析时返回了明确错误

这个状态会驱动表格里的图标和 tooltip。

## 重命名模板与持久化设置

重命名规则本质上是一段字符串模板。支持的变量定义在 `src/const/index.ts`：

- 日期变量：`{YYYY}`、`{MM}`、`{DD}`、`{hh}`、`{mm}`、`{ss}`
- 通用元数据/日期：`{Date}`、`{Make}`、`{Camera}`
- 图片元数据：`{Lens}`、`{FocalLength}`、`{Aperture}`、`{Shutter}`、`{ISO}`
- 当前文件名变量：`{Current}`、`{current}`

`useInputFormat` 负责可见输入值、生效模板和模板历史记录。`OperationBar` 负责具体输入事件。

输入框会先去掉首尾空格，再更新实际生效的模板。中文输入法等 IME 组合输入期间，模板更新会延后到组合结束后。

执行重命名前没有单独的模板校验步骤：

- 生效模板为空时，预览会把所有文件标记为跳过；点击“重命名”后只会显示“无需执行重命名操作”的 toast，不会调用后端
- 模板普通文本或元数据替换值里的文件名非法字符会在预览生成阶段被移除，而不是阻止重命名

前端会把几类用户设置存到 local storage：

- 语言
- 严格模式
- 使用创建日期
- 模板历史记录

默认设置：

- `strictMode`：关闭
- `useCreatedDate`：关闭
- `language`：优先使用已保存值；没有保存值时，浏览器语言以 `zh` 开头则使用中文，否则使用英文
- `format history`：`['{YYYY}{MM}{DD} {hh}.{mm}.{ss}']`

模板历史最多保留最近 5 个去重后的值，并显示在输入框的历史下拉中。历史记录只会在重命名成功后更新，不会在每次编辑输入框时更新。

## 预览文件名的生成逻辑

预览完全在前端通过 `transformIPCFiles` 生成。

### 排序

- 在生成预览前，文件会先按原文件名做带数字感知的排序。

### 基础 UI 转换

前端会先为每个文件计算：

- 当前路径和目录
- 原文件名
- 可读的创建时间
- 可读的文件大小
- 元数据状态和提示文案

### 严格模式

如果当前生效模板为空，预览会把所有文件标记为跳过，并保留原文件名。

如果开启严格模式，当当前模板所需字段无法被该文件满足时，这个文件会被跳过。

例子：

- 模板用了 `{Date}`，且没有启用“使用创建日期”，那么没有 metadata date 的文件会被跳过。
- 模板用了 `{Lens}`，但文件没有 lens 元数据，也会被跳过。

被跳过的文件会保留原文件名，并且不会进入最终的重命名请求。

如果关闭严格模式，变量缺失不会导致跳过。缺失值会被替换成可读占位文本，例如 `YYYY`、`Date`、`Make`、`Lens`、`ISO`。

### 使用创建日期

如果开启 `useCreatedDate`，日期相关变量将优先使用文件创建时间，而不是元数据里的拍摄时间。日期相关变量包括拆分后的日期变量和 `{Date}`。

后端传给前端的是毫秒级文件创建时间戳。如果当前平台无法提供创建时间，后端会先回退到 Unix epoch，再由前端按本地时区格式化。

### 非法字符清洗

`generateFilename` 会在使用前清洗模板普通文本和变量替换值。

当前清洗逻辑会移除以下字符：

- `\ / : * ? " < > |`

这意味着预览结果可能和用户输入的文本不完全一致。例如，普通文本片段 `a:b` 会变成 `ab`。当前不会因为文件名非法字符单独弹出重命名错误。

### 扩展名处理

重命名模板会被当作目标 basename。`generateFilename` 总会追加原文件名里的扩展名。

具体影响：

- 用户不应该在模板里手写扩展名
- 如果模板是 `archive`，原文件是 `IMG_0001.JPG`，预览目标会是 `archive.JPG`
- 如果模板是 `archive.jpg`，原文件是 `IMG_0001.JPG`，预览目标会是 `archive.jpg.JPG`

### 已加载集合内的重名处理

前端只会在“当前已经加载到应用里的文件集合”内检测目标文件名是否重复。

行为如下：

- 比较时不区分大小写。
- 如果多个已加载文件计算出了同一个目标文件名，应用会自动追加 `_1`、`_2` 这类序号；序号会按重名组大小补齐位数，因此 10 个重名目标会得到 `_01` 到 `_10`。
- 重名组里的每个未跳过文件都会追加序号，包括该组里的第一个文件。
- 序号会插入到“生成后的预览文件名”的扩展名式片段之前。正常使用时，这个扩展名就是原文件扩展名，因为 `generateFilename` 会把原扩展名追加到生成 basename 后。
- 如果某个文件最终预览文件名与原文件名完全一致，它会被标记为 `shouldSkip`。
- 严格模式下被跳过的文件，仍会用它的原文件名参与重名计数，这样需要重命名的文件不会撞上将保持不变的文件。

这里有一个关键限制：

- 这套重名处理不会把磁盘上“存在但当前没有加载到 UI 中”的文件考虑进去。
- 如果一个目录里总共有 10 个文件，但用户只拖进来了 5 个，那么预览阶段的去重只会看到这 5 个。
- 重名检测不会按目录分组。如果来自不同目录的已加载文件计算出相同文件名，前端仍会把它们当作重复文件名并追加序号。
- 当前实现里的内部 `nameMap` 使用原文件名作为 key，而不是完整路径。如果来自不同目录的两个已加载文件拥有相同原文件名，它们的预览映射可能互相覆盖。这个点应视为当前实现限制，而不是建议依赖的预期行为。

## 执行重命名的流程

当用户点击“重命名”时：

1. 前端筛出所有 `shouldSkip === false` 的文件。
2. 组装成 `{ oldPath, newFilename, tempFilename }[]` 结构。
3. `oldPath` 使用后端返回的 `pathname`。
4. `newFilename` 使用前端预览生成的新文件名。
5. `tempFilename` 使用 `<timestamp>_<originalFilename>`。
6. 把这批数据传给后端 `rename_files`。

前端不再构造完整目标路径。后端会先校验 `newFilename` 和 `tempFilename` 都是普通文件名，然后用 Rust `Path::join` 把它们解析到 `oldPath` 的父目录下。

如果当前已经在重命名中，后续点击会被忽略。如果没有任何文件需要重命名，前端会显示“无需执行重命名操作”的 toast，并且不会调用后端。

## 与现有文件冲突时的处理

后端会先把每个 payload 项转换成具体的重命名计划：

- `old_path` 来自 `oldPath`
- `new_path` 是 `parent(oldPath).join(newFilename)`
- `temp_path` 是 `parent(oldPath).join(tempFilename)`

然后在真正执行任何 rename 之前，做一次预检查。

对于每一个 `new_path`：

- 如果这个路径同时也是本批次里某个文件的 `old_path`，后端会放行，因为两阶段重命名会先把它腾空。
- 否则，只要 `new_path` 在磁盘上已经被占用，后端就会立刻返回 `AlreadyExists` 错误。

对于每一个 `temp_path`：

- 只要这个路径在磁盘上已经被占用，后端就会立刻返回 `AlreadyExists` 错误。
- 这个检查比 `new_path` 更严格，因为 phase 1 会在任何批次内 `old_path` 被安全释放之前先写入临时路径。

预检查使用能识别符号链接自身的 metadata 读取方式，因此“被占用”的路径包括普通文件、目录、符号链接和 dangling symlink。

后端当前也不会对 payload 内部重复的最终 `new_path` 做去重检查；它依赖前端预览生成阶段避免重复目标。路径是否相同基于派生出的 `PathBuf` 判断，不会先做 canonicalize。

这正是“只拖入部分文件”场景下的关键行为：

- 前端预览阶段并不知道那些没有加载进来的文件。
- 后端执行阶段会检查真实文件系统。
- 如果被拖入的文件想改成一个已经被“未拖入文件”占用的名字，那么这次重命名会失败。

当前行为是“整批失败”，不是“部分成功”：

- 后端在 phase 1 之前就会先做存在性检查。
- 只要发现一个冲突，本批次不会有任何文件开始重命名。
- 前端会把后端返回的错误信息展示在错误弹窗里。

前端错误统一经过 `useError`，并使用 Tauri 的原生 `message` 对话框展示。字符串错误会直接显示；如果传入对象带有 `Error.message`，则显示该 message；否则使用本地化的系统错误文案。

## 两阶段重命名

如果预检查通过，后端会执行两阶段重命名：

1. `oldPath -> tempPath`
2. `tempPath -> newPath`

这样设计的目的：

- 避免多个文件互换名称时直接相撞。
- 允许目标路径当前暂时被“同一批次里的另一个旧文件名”占用。

这个设计可以处理常见的批量改名碰撞，但它本质上仍然是文件系统操作，不是完整事务系统。

如果 phase 1 或 phase 2 在部分文件已经改名后失败，后端当前会返回文件系统错误，但不会回滚之前已经完成的 rename 操作。

## 重命名成功后的刷新

后端成功后会返回新的路径列表。前端随后会：

- 用“未改名的文件 + 已改名后的新路径”重新构建当前文件列表
- 这次重建会走 `handleDropFiles`，因此合并后的路径会通过后端的显式路径命令重新加载
- 如果当前选中的文件刚好被改名了，尝试把选中项映射到新路径
- 显示成功 toast
- 把当前模板写回历史记录

## UI 展示规则

表格会展示：

- 原文件名
- 预览中的新文件名
- 元数据状态图标

如果某个文件被标记为 `shouldSkip`，那么“新文件名”这一列会显示为空。

右侧详情面板会展示：

- 尽可能可预览的图片内容
- 对小于 `50_000_000` 字节的视频，会用同一个图片元素尝试加载 asset URL；当前没有视频播放器，也没有生成视频缩略帧的步骤
- 基础文件信息
- 已提取的元数据字段

当前预览规则：

- 图片可以预览
- 视频只有在小于 `50_000_000` 字节时才尝试加载；如果图片加载失败，会回退到通用文件图标
- 其他文件显示通用图标

设置面板打开时，拖拽事件会被忽略。

## 实际总结

这个项目把“预览逻辑”和“真实文件系统安全检查”明确拆开了：

- 前端负责基于当前已加载文件生成用户可见的重命名预览
- 后端负责在最终执行时阻止覆盖磁盘上的无关文件

因此，“预览去重”和“最终冲突处理”是两个相关但不相同的层次：

- 前端去重解决的是当前已加载集合内部的冲突
- 后端冲突检查阻止的是对磁盘上其他现有文件的覆盖

这就是为什么某一批文件在 UI 预览里看起来没有问题，但在真正执行重命名时仍可能失败，尤其是在用户只加载了目录里一部分文件的时候。
